# WireGrid Vitrine — Developer Guide

## Project Overview

Marketing site for WireGrid (the SCADA/IIoT product), live at https://wiregrid.fr. Pure static Astro build, deployed via Cloudflare Workers (wrangler). The same Worker also hosts the trial license signer endpoint (`POST /api/trial`) used by the WireGrid product. See cross-repo references below.

## Stack

| Layer | Choice |
|-------|--------|
| Build | Astro 6 (static build into `dist/`) |
| Edge runtime | Cloudflare Workers via wrangler 4 (`src/worker.js`) |
| CSS | Tailwind v4 |
| i18n | French (default) + English. Routes `/<lang>/`. |
| Tests | vitest |

The Worker is currently a 14-line proxy of `env.ASSETS`. Vague 5 will turn it into a proxy plus a `POST /api/trial` route. Stay minimal: do not pile additional logic on the edge unless explicitly scoped.

## Branches and deploy

| Branch | Purpose | Auto-deploy |
|--------|---------|-------------|
| `master` | Prod, serves `wiregrid.fr` and `www.wiregrid.fr` | YES (`.github/workflows/deploy.yml`, runs after CI green on master) |
| `dev` | Integration, accumulates work | NO |

**Never push to `master` without explicit user approval.** Master push triggers a Cloudflare Workers deploy to prod after CI completion. Default workflow: PR `dev` to `master`, the user merges when they want a release. Hotfix path: branch from `master`, PR back to `master`, then merge `master` into `dev` to stay aligned.

## Hard rules

- **No em-dashes (long horizontal hyphen)** in code, copy, comments, emails, JSON values. Use colons, parentheses, periods. Applies in all languages, all files.
- **No competitor names in marketing copy**: Niagara, Ignition, EcoStruxure, Tridium, Inductive Automation, Workbench, JACE. Refer to the category ("legacy SCADA platforms", "tag-priced SCADA", etc.).
- **Never write the trial signing private key to disk**, even temporarily, even in tests or fixtures. Read it from `env.TRIAL_SIGNING_KEY` at runtime only. The user's Bitwarden vault is the only canonical store.
- **Never frame shortcuts as laziness.** State the real tradeoff (time-boxing, dependency wait, blast-radius minimization).
- **No phone-home in product traffic.** The `/api/trial` Worker ends with the Resend send. The verifier in the sister repo runs offline. Telemetry, if added later, is opt-in and non-blocking. Do not slip in anonymous pings.

1. New work: branch from `dev`, code, test locally, push the feature branch, PR to `dev`, merge after CI green.
2. Release: PR `dev` to `master` with a release-style title. The user merges; the Worker redeploys automatically.
3. Cross-repo coordination on the trial system: the user is the relay. He pastes a short message in your kickoff or in the conversation when keys, schema, or task scope change on the wiregrid side. You do not poll the other repo.

## Quickstart

```bash
npm install                    # one-shot
npm run dev                    # local Astro dev server (no Worker logic)
npm test                       # vitest
npm run build                  # produces dist/
npx wrangler dev               # local Worker (build dist/ first)
```

To exercise `/api/trial` locally once the route is implemented:

```bash
npm run build && npx wrangler dev
# in another shell:
curl -X POST http://localhost:8787/api/trial \
  -H 'content-type: application/json' \
  -d '{"email":"test@example.com","name":"Test","turnstileToken":"..."}'
```

## Key files

- `src/worker.js` Cloudflare Worker entry (Vague 5 extends it with `/api/trial`)
- `wrangler.jsonc` Worker config, custom domains, bindings (KV, rate limiting, secrets)
- `src/pages/<lang>/index.astro` landing pages (FR default, EN)
- `src/pages/<lang>/trial.astro` trial signup form (Vague 5, à créer)
- `src/components/` Astro components (Hero, Features, Pricing, FAQ, etc.)
- `src/i18n/` translation keys
- `tests/built-html.test.ts` smoke test on the built static output
- `.github/workflows/ci.yml`, `.github/workflows/deploy.yml` CI + auto-deploy
